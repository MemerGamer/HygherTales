# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

### Development

```bash
# Full dev (both proxy + desktop, with shared built first)
bun run dev:all

# Proxy only (hot reload, serves at http://localhost:8787)
bun run dev:proxy

# Desktop only (Tauri dev with Vite frontend)
bun run dev
```

### Build

```bash
# Build everything (shared → proxy → desktop)
bun run build

# Build sidecar binary for desktop distribution (requires CURSEFORGE_API_KEY)
cd apps/proxy && bun run build:sidecar -- --target=linux|windows|macos
```

### Lint & Format

```bash
bun run lint
bun run format
bun run format:check
```

### Tests (proxy only)

```bash
cd apps/proxy && bun test src
```

### Shared package (must rebuild after schema changes)

```bash
cd packages/shared && bun run build
```

## Setup

Before running for the first time:

```bash
cp apps/proxy/.env.example apps/proxy/.env
# Set CURSEFORGE_API_KEY and CURSEFORGE_GAME_ID=70216
```

## Architecture

This is a Bun monorepo with three packages:

### `packages/shared`
Zod schemas and TypeScript types shared between proxy and desktop. **Must be built (`bun run build`) before the other packages can consume it.** All API response shapes are defined here.

### `apps/proxy`
Hono/Bun HTTP server that proxies CurseForge and Orbis.place APIs. The CurseForge API key lives here only — the desktop never holds it. In production (desktop distribution), this is compiled to a standalone Bun binary (sidecar) via `build-sidecar.ts`, which **embeds the API key** at compile time into `src/.embedded-env.ts`.

- Routes under `/v1/`: search, featured, categories, mod details/files, download URLs, Orbis.place
- In-memory cache (60s TTL), per-IP rate limiting
- Proxy env vars: `CURSEFORGE_API_KEY`, `CURSEFORGE_GAME_ID`, `PORT` (8787), `CORS_ORIGINS`, `RATE_LIMIT_PER_MIN`

### `apps/desktop`
Tauri v2 app with a React/Vite frontend (Tailwind v4).

**Boot sequence:** Rust `setup` hook auto-starts the sidecar → frontend polls `GET /health` until ready → app renders. The frontend never calls `invoke("start_proxy_sidecar")`; the sidecar is always managed by Rust.

**Tauri commands** (all in `apps/desktop/src-tauri/src/lib.rs`): file system operations (mod install/move/delete/update, ensure dirs, download), mod/profiles JSON persistence in app data dir, game launching, path detection, sidecar lifecycle.

**Frontend** (`apps/desktop/src/`):
- State-machine page navigation in `App.tsx` (no router): `home | browse | installed | settings`
- All API calls to the proxy go through `lib/api.ts`, validated against shared Zod schemas
- Settings (proxy URL, mods dir, game exe) persisted in `localStorage` via `lib/settings.ts`
- Installed mods and profiles are stored as JSON files in the Tauri app data directory via Rust commands (`read_installed_mods`, `write_installed_mods`, `read_profiles`, `write_profiles`)

## Key Constraints

- The sidecar binary must be at `apps/desktop/src-tauri/binaries/hyghertales-proxy-<triple>` before running `tauri build`
- `src/.embedded-env.ts` in the proxy is auto-generated by `build-sidecar.ts` — do not edit manually
- The shared package `dist/` must exist before building proxy or desktop TypeScript
- Settings override the proxy URL at runtime; default is `http://localhost:8787`
